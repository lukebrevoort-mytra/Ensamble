# Workflow Kit

Three portable, **dynamic** software-development workflows that adapt to whatever
repo they're dropped into — `/spec`, `/execute`, `/review` — plus a shared
contract and an installer that retrofits them to a specific repo.

The design goal: **no brittle static checklists, no sprawling library of role
files.** The workflows inspect the repo at runtime, generate the specialists they
need on the fly, distinguish proof from guesswork, and revise themselves when
reality diverges. This repo is the **canonical source**; `/workflow-install`
copies the portable layer into any target repo and derives a repo-specific profile.

## Layout

```
workflow-kit/                     ← this repo (canonical source of truth)
  CONTRACT.md                     shared rules + output format (generic, identical everywhere)
  commands/
    spec.md  execute.md  review.md    the three workflow prompts
    workflow-install.md           the installer/retrofit command
  templates/
    repo-profile.template.md      skeleton the retrofit fills in per repo
  docs/
    architecture.html             house-style architecture overview (open in a browser)
```

Running `/workflow-install` in a target repo produces the **per-repo** layout:

```
<repo>/.claude/                   ← committed; shared with the team
  workflow-kit/
    CONTRACT.md                   (copied verbatim — portable layer)
    repo-profile.md               (generated by retrofit — repo-specific layer)
  commands/
    spec.md  execute.md  review.md    → /spec /execute /review in this repo
<repo>/.workflows/                ← gitignored scratch / handoff
    recon.md                      cached repo profile (all 3 workflows reuse)
    spec-<slug>.md  review-<slug>.md   artifacts for handoff across sessions
```

## The three layers (what's static vs dynamic)

| Layer | Files | Per-repo? | Lifetime |
|---|---|---|---|
| **Portable** | `CONTRACT.md` + 3 command prompts | No — identical everywhere | Committed |
| **Repo-specific** | `repo-profile.md` (incl. derived **specialist roster**, **invariants→gate-tests**, **execution mode**) | Yes — generated by retrofit, then maintained | Committed |
| **Dynamic / ephemeral** | `recon.md`, per-task artifacts, runtime subagent roles | Per run | Gitignored / in-memory |

The portable layer never names a language or tool. Everything repo-specific lives
in `repo-profile.md` (human-confirmed) or `recon.md` (auto-detected). Specialist
reviewers/executors are **authored at runtime** as Agent prompts — there is no
permanent role library to maintain. See `docs/architecture.html` for the full picture.

## How they stay portable & adaptive

- **Recon-first:** detect language, layout, test framework, and canonical commands
  from manifests/CI before acting (CONTRACT §1), then cache it.
- **Profile over detection:** the retrofit *reads the architecture docs* and derives
  this repo's character → a **specialist roster**, **non-negotiable invariants paired
  with their exact gate tests**, and a **characteristic execution mode**. A sim-heavy
  k8s repo and a data-UI repo come out with different rosters and acceptance models.
- **Evidence discipline:** every claim is tagged FACT ✓ / ASSUMPTION ~ / QUESTION ? /
  BLOCKED ⛔ (CONTRACT §2) — proof and guesswork never blur.
- **Dynamic readjustment:** defined triggers force an observe→reframe→re-plan→log
  loop (CONTRACT §4) when the repo, spec, coupling, or test path surprises you.
- **One output shape:** all three workflows report in the CONTRACT §5 format.

## Install

1. **Make `/workflow-install` available globally** (so you can run it in any repo).
   Symlink (recommended — stays in sync with this repo) or copy:
   ```
   ln -sf ~/work/mytra/workflow-kit/commands/workflow-install.md \
          ~/.claude/commands/workflow-install.md
   ```
2. **In a target repo**, run `/workflow-install`. It copies the portable layer into
   `<repo>/.claude/`, runs recon, reads the design docs, interviews you, and writes
   `repo-profile.md`. The repo then has `/spec`, `/execute`, `/review`.

## Usage

```
/workflow-install        # once per repo — installs + retrofits to this repo
/spec     <idea|ticket>  # idea → implementation-ready spec (writes .workflows/spec-*.md)
/execute  <spec path>    # spec → adaptive TDD implementation until criteria pass
/review   [pr|branch]    # diff → risk-driven review with verdict + evidence
```

`/spec` → `/execute` → `/review` chain via the `.workflows/` artifacts. Re-running
`/workflow-install` updates the portable layer without touching `repo-profile.md`.
