Portable · Dynamic · Evidence-first

A repo that teaches the workflow, not a workflow that assumes the repo.

Three reusable software-development workflows — /spec, /execute, /review — that inspect the repository at runtime, generate the specialists they need on the fly, separate proof from guesswork, and revise themselves when reality diverges. Five small files. No static checklists, no permanent role library.

01Three layers

Everything in the system sorts into exactly one of three layers. This is the answer to "what is static vs dynamic": the portable layer never names a tool, the repo layer is generated once and maintained, and the dynamic layer is reborn every run.

Portable · committed · identical everywhere

The contract & the three prompts

CONTRACT.md holds the operating rules and the single output format. The three command prompts hold each workflow's goal and phases. None of these name a language, framework, or path — so they drop into any repo unchanged.

The repo profile

repo-profile.md is the specialization layer — written by the retrofit interview. It pins the repo's real commands, test & simulation harnesses, the MCP services agents may use, subsystem boundaries, and the merge bar. This is what makes generic agents repo-aware.

Recon, artifacts & runtime roles

.workflows/recon.md caches detected repo facts. Per-task specs and review reports hand off between workflows. Specialist reviewers and executors are authored as prompts at runtime — never stored as files. Nothing here is committed.

02The three workflows

Each is a thin prompt that loads the contract + profile, then runs adaptive phases. They chain through the gitignored .workflows/ handoff space.

/spec

Spec

Idea → implementation-ready spec, grounded in real code.

  1. Orient — recon & canonical commands
  2. Gather context — parallel explorers, find patterns
  3. Define — criteria, affected areas, risks, test strategy
  4. Report & hand off
→ .workflows/spec-<slug>.md
/execute

Execution

Spec → adaptive TDD build until criteria & checks pass.

  1. Orient & lock acceptance criteria
  2. Decompose into verifiable tasks
  3. Implement — TDD where a test path exists
  4. Readjust on hidden coupling / blocks
  5. Converge — loop until green, then verify
→ task ledger + evidence table
/review

PR Review

Diff → risk-driven review with a verdict & evidence.

  1. Acquire diff (PR / branch / merge-base)
  2. Infer risk → choose lenses
  3. Spawn specialist reviewers in parallel
  4. Run the checks that matter
  5. Verify findings adversarially → verdict
→ .workflows/review-<slug>.md

03Evidence & dynamic roles

Two mechanisms do the heavy lifting: a tagging discipline that keeps proof distinct from narrative, and runtime role generation that replaces a permanent agent library.

Every claim wears a tag

FACT ✓verified by command output, file:line, or a test — cited inline
ASSUMPTION ~plausible, unverified — notes how to verify & what breaks if wrong
QUESTION ?needs a human decision — notes what unblocks it
BLOCKED ⛔can't proceed — notes the blocker & workarounds tried

An assumption never graduates to fact without evidence. The reader can always tell what was proven.

04Dynamic readjustment

The workflows expect to be wrong up front. Defined triggers force a visible course-correction loop — silent changes aren't allowed; every readjustment lands in the report.

Observe — expected vs found→ Reframe — what changes→ Re-plan — smallest delta→ Log — Readjustments section↺
  • repo structure differs from the plan's assumption
  • the spec is incomplete or self-contradictory
  • implementation reveals hidden coupling
  • the intended test path is unavailable → fall back to characterization / smoke check
  • a verification step is blocked → record BLOCKED, gather best alternative evidence
  • a PR's real risk profile differs from the first read → re-scope reviewers
