Native orchestration · Repo-aware · Evidence-first

A repo that teaches the workflow, not a workflow that assumes the repo.

/spec, /execute, and /review are thin launchers built on Claude's native Workflow engine. Each gathers repo context (profile + recon) and injects it as static args into a portable, schema-validated workflow script — because the script's sandbox can't read the repo, but the agents it spawns can. Every spawned agent re-orients on that context, uses the repo's real tools, and produces structured evidence. Mandatory requirement gates — declared once at install — ensure the workflow never calls something done without the proof.

01Four layers

Everything in the system sorts into exactly one of four layers. The first two are portable and identical everywhere. The third is generated per-repo at install. The fourth is what the user types — thin launchers that wire the others together.

Layer 1 · Guideline · Portable · identical everywhere

The orchestration contract

CONTRACT.md is the shared operating standard: the launch model (thin launcher → native workflow → structured report), the sandbox truths, the standard agent brief that prevents naked subagents, the tool-awareness rule, adaptive scale, adversarial-verify, and the canonical output schemas. Nothing here names a language, framework, or repo.

Layer 3 · Adjustment · Per-repo · generated at install

Repo profile & mandatory requirements

repo-profile.md is the specialization layer written by /ensemble-install. It pins the roster, canonical commands, invariants → gate tests, agent types, and the repo tools agents must use. Critically, it captures mandatory requirements — install-time gates like "any UI change must produce a browser screenshot" — that the scripts read as args.mandatoryRequirements and enforce via a verification loop.

Layer 4 · Entry points · Portable · thin launchers

Command launchers

The /spec, /execute, and /review commands run in the main agent. Each reads the contract and profile, ensures recon is fresh, resolves the target, gathers changed files and canonical commands as FACTs, then calls Workflow({name, args}). The command is what authorises the Workflow tool's opt-in. After the script returns, the command renders the structured result into the §6 report and saves the artifact.

02The three workflows

Each is a native workflow script invoked via Workflow({name, args}). The launcher injects repo context; the script orchestrates deterministic fan-out and returns a structured object the command renders. Artifacts chain through the gitignored .workflows/ handoff space.

/spec

Spec

Request → implementation-ready spec grounded in real code and existing patterns.

  1. Scope — identify the areas and patterns the request touches
  2. Gather — parallel explorers per area return file:line context and patterns to mirror
  3. Draft — synthesize a testable spec anchored to existing code
  4. Critique — adversarially check for gaps, contradictions, untestable criteria
→ .workflows/spec-<slug>.md
/execute

Execution

Spec → adaptive TDD build; loops until all criteria and mandatory evidence pass.

  1. Plan — decompose spec into small, independently verifiable tasks
  2. Implement — implement each open task (TDD where a test path exists)
  3. Verify ↺ — verifier re-checks each task and its mandatory evidence; misses send work back to Implement
  4. Checks — final canonical checks + invariant gate tests + mandatory-requirements gate
→ task ledger + evidence table
/review

PR Review

Diff → risk-driven specialist review with adversarial verify and a structured verdict.

  1. Shape — map the change (intent, structure, reading order, hotspots) and triage risk lenses + invariants in blast radius
  2. Review — one specialist per matched roster role or risk lens (parallel)
  3. Verify — adversarially refute each finding; drop ones that don't survive
  4. Checks — canonical build/typecheck/lint/tests + invariant gates + mandatory-requirements gate → verdict
→ .workflows/review-<slug>.md

03Evidence & the standard brief

Two mechanisms do the heavy lifting: a tagging discipline that keeps proof distinct from narrative, and a shared brief that ensures every spawned agent is oriented on repo context and knows its tools — no naked subagents.

Every claim wears a tag

FACT ✓verified by command output, file:line, or a test — cited inline
ASSUMPTION ~plausible, unverified — notes how to verify & what breaks if wrong
QUESTION ?needs a human decision — notes what unblocks it
BLOCKED ⛔can't proceed — notes the blocker & workarounds tried

An assumption never graduates to fact without evidence. Structured output schemas (Finding / Verdict / Checks / Requirements) encode this discipline at the tool layer — the model retries on schema mismatch.

No naked subagents — the standard brief

Every agent() prompt is built through a shared brief(): role → orient on in-scope files + injected profile/recon → use repo tools via ToolSearch → single job → evidence discipline → schema note. Roster roles (from the profile) are spawned first by their real agentType; generic lenses fill the gaps:

The repo roster always goes first. Generic lenses only cover risk the roster misses. A UI lens gets the browser MCP; a data lens gets the DB MCP — tool-awareness from the profile, not guessing.

04Dynamic readjustment

The workflows expect to be wrong up front. Defined triggers force a visible course-correction loop — silent changes aren't allowed; every readjustment lands in the report.

Observe — expected vs found→ Reframe — what changes→ Re-plan — smallest delta→ Log — Readjustments section↺
  • repo structure differs from the plan's assumption
  • the spec is incomplete or self-contradictory
  • implementation reveals hidden coupling or an undocumented dependency
  • the intended test path is unavailable → fall back to characterization / smoke check
  • a verification step is blocked → record BLOCKED, gather best alternative evidence
  • a PR's real risk profile differs from the first read → re-scope reviewers
  • mandatory-requirement evidence is missing → work re-enters the implement→verify loop until it exists or is BLOCKED ⛔

05Install & retrofit

/ensemble-install drops the portable layer into a repo, runs detection, and interviews the user so the scripts know this repo's commands, harnesses, tools — and its non-negotiables.

Interview

Profile

06On disk

The committed footprint stays small. The portable layer (contract + scripts + launchers) is identical everywhere; the per-repo profile is the only customisation. Scratch stays gitignored.
